# Field Types — Payload Format for Record Creation

## Type → Payload Mapping

| Field type | Payload value | Notes |
|------------|--------------|-------|
| `text` / `url` / `server` | `"string value"` | Plain string |
| `richText` | `"<p>HTML content</p>"` | Accepts HTML markup |
| `number` / `percentage` | `123` or `12.5` | Numeric |
| `boolean` | `true` / `false` | |
| `autoNumber` | **do not send** | Auto-generated by the server |
| `date` | `"2026-03-16"` | ISO 8601 date string |
| `dateTime` | `"2026-03-16T14:00:00.000Z"` | ISO 8601 with time (UTC) |
| `picklist` (single, `maxSelected=1`) | `"option_key"` | String key from `options` |
| `picklist` (multi, `maxSelected>1`) | `["key1", "key2"]` | Array of string keys |
| `lookup` / `inheritLookup` | `{ "_id": "record-id" }` | Send only `_id`; server fetches description fields |
| `lookup` (clear) | `{}` | Empty object clears the lookup |
| `email` | `{ "address": "a@b.com" }` | Optional `"type"` field |
| `phone` | `{ "countryCode": 55, "phoneNumber": "5511999999999" }` | Full number incl. country + area code |
| `personName` | `{ "first": "João", "last": "Silva" }` | Server computes `full` automatically |
| `address` | `{ "country": "BRA", "state": "SP", "city": "...", "place": "...", "number": "..." }` | |
| `money` | `{ "currency": "BRL", "value": 100.0 }` | |
| `geoloc` | `[-51.22, -30.04]` | `[longitude, latitude]` |
| `json` | any object or array | Stored as-is |
| `file` | `{ "key": "...", "name": "...", "size": 1234 }` | |

---

## Picklist Rules

- **Single-select** (`maxSelected = 1`): send a string → `"Ativo"`
- **Multi-select** (`maxSelected > 1`): send an array → `["Alta", "Média"]`
- **Required multi-select** (`minSelected > 0`): array must not be empty
- Values must exactly match the **keys** in the field's `options` (case-sensitive)
- An empty string `""` is treated as `null` (not provided)

### How to get valid picklist options

```bash
python3 skills/konecty-modules/scripts/modules.py fields "<Module>"
```

The options column shows the valid keys. Example for `Message.status`:

```
status  picklist  Situação  options: Nova, Rascunho, Enviada, Send, Falha no Envio…
```

So valid values are: `"Nova"`, `"Rascunho"`, `"Enviada"`, `"Send"`, `"Falha no Envio"`, etc.

---

## Lookup Rules

1. **Always send `{ "_id": "<record-id>" }`** — never send label or code directly.
2. The server validates the `_id` exists in the related document's collection.
3. If not found → `"Record not found for field {fieldName} with _id [{id}] on document [{Document}]"`
4. The server automatically copies `descriptionFields` (e.g. `code`, `name.full`) into the parent record.

### How to get a lookup _id

Use the `lookup` subcommand to find the `_id` by code or name:

```bash
# By numeric code
python3 skills/konecty-create/scripts/create.py lookup Contact 16503

# By name / text
python3 skills/konecty-create/scripts/create.py lookup Campaign "Black Friday"

# With extra display fields
python3 skills/konecty-create/scripts/create.py lookup Contact 16503 --fields "code,name,email"
```

Or use `konecty-find` for more complex queries:

```bash
python3 skills/konecty-find/scripts/find.py query Contact \
  --filter '{"match":"and","conditions":[{"term":"code","operator":"equals","value":16503}]}' \
  --fields "_id,code,name"
```

## Required Fields

The API does not expose required field flags in the field listing. Required fields are enforced at save time. If a required field is missing, the API returns:

```json
{ "success": false, "errors": [{ "message": "Field {fieldName} is required" }] }
```

**Strategy:** Start with the fields you know. If the API returns required-field errors, add those fields and retry.

Fields with `defaultValue` defined in the module metadata are automatically populated by the server before the required check, so they do not need to be sent explicitly.

---

## Null / Empty Values

- `null` values in the payload are **stripped** before processing — equivalent to not sending the field.
- `""` (empty string) is treated the same as `null`.
- To explicitly **clear a lookup**, send `{}` (empty object).

## API Response

### Success

```json
{
  "success": true,
  "data": [
    {
      "_id": "generated-id",
      "subject": "Bem-vindo!",
      "status": "Nova",
      "_createdAt": "2026-03-16T15:00:00.000Z",
      "_createdBy": { "_id": "...", "name": "..." }
    }
  ]
}
```

`data` is always an array with one record. The record reflects the server-normalized values (defaults applied, computed fields populated).

### Error

```json
{
  "success": false,
  "errors": [
    { "message": "Field subject is required" },
    { "message": "Record not found for field contact with _id [xyz] on document [Contact]" },
    { "message": "[Message] You don't have permission to create field status" }
  ]
}
```

HTTP status is always `200` — success/failure is in the `success` field.

---

## Common Error Messages

| Error | Cause | Fix |
|-------|-------|-----|
| `Field {name} is required` | Required field missing | Add the field to the payload |
| `Record not found for field {name} with _id [x] on document [Y]` | Lookup `_id` doesn't exist | Use `create.py lookup` to get the correct `_id` |
| `Value {x} for field {name} is invalid` | Picklist value not in options | Use `konecty-modules fields` to check valid options |
| `[Module] Data must have at least one field` | Empty payload | Send at least one field |
| `You don't have permission to create field {name}` | Field not creatable by current user | Omit that field; the server will use the default or leave it empty |
| `[Module] You don't have permission to create records` | Module not accessible | Check session and user permissions |
